========================================
SETUP INSTRUCTIONS
========================================

STEP 1: Create Users in Supabase Auth
--------------------------------------
Go to Supabase Dashboard > Authentication > Users
Create each user manually OR use the Supabase Admin API:

Using Supabase CLI (recommended):
```bash
# Install Supabase CLI if needed
npm install -g supabase

# For each user, create via Supabase Dashboard or API
```

Or use the Supabase Dashboard:
1. Navigate to Authentication > Users
2. Click "Add User" or "Invite User"
3. Enter email and password for each user
4. Note down the UUID for each user

STEP 2: Update SQL File
--------------------------------------
1. Open sample_data_import.sql
2. For each user, replace the placeholder UUID with the actual UUID from auth.users
3. The UUID format should match: 'xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx'::uuid

STEP 3: Run SQL Import
--------------------------------------
1. Open Supabase Dashboard > SQL Editor
2. Copy and paste the contents of sample_data_import.sql
3. Replace all UUID placeholders with actual auth.users.id values
4. Run the SQL script
5. Verify users and recipes were created successfully

ALTERNATIVE: Using Supabase Admin API
--------------------------------------
You can also create users programmatically using the Supabase Admin API.
This requires your service role key (keep it secret!).

Example using curl:
```bash
curl -X POST 'https://YOUR_PROJECT.supabase.co/auth/v1/admin/users' \
  -H "apikey: YOUR_SERVICE_ROLE_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "email": "sarah.chen@example.com",
    "password": "TestPassword123!",
    "email_confirm": true
  }'
```

Then get the user ID from the response and use it in the SQL file.

========================================
TROUBLESHOOTING
========================================

Issue: Foreign key constraint errors
Solution: Make sure you've created users in auth.users first and are using the correct UUIDs

Issue: RLS policy blocking inserts
Solution: Run the SQL script as a service role or temporarily disable RLS for testing

Issue: Duplicate email/username errors
Solution: Clear existing test data or use different emails/usernames
